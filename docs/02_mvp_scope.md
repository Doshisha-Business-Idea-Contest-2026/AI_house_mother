# 02. MVP スコープ

## 1. このドキュメントの目的

本ドキュメントは AI 寮母 MVP の**スコープ**を確定するためのものである。

- MVP に**含む**機能
- MVP から**除外する**機能（今後の展望として発表資料には残す）
- スコープ変更が必要な場合の**優先順位**
- MVP の**成功基準**

要件の全体像は `01_requirements.md`、実装の具体は `04_functional_spec.md` を参照。

## 2. スコープ確定の原則

### 2.1 判断基準

MVP は「決勝プレゼンで実演でき、審査員に価値が伝わる」ことを最優先とする。以下の順で優先度を判断する。

1. **デモシナリオ（`08_demo_scenario.md`）に登場する機能** — 必須
2. **プロダクトの中心コンセプトを体現する機能** — 必須
3. **AI 寮母の差別化を示す機能**（先輩投稿の活用、SECI モデル体現） — 推奨
4. **ステークホルダー価値を補強する機能** — 余裕があれば
5. **将来性・拡張性を示す機能** — 除外、発表資料で言及のみ

### 2.2 スコープ変更ルール

MVP 実装期間中にスコープを変更する場合は、以下の手順を踏む。

1. 変更理由を明記する（時間不足、技術的難易度、優先度見直し等）
2. `.codex/rules/development_policy.md` の意思決定記録方針に従い、判断を残す
3. 本ドキュメントの該当セクションを更新する
4. 変更履歴セクションに記録する

## 4. MVP から除外する機能

以下は「今後の展望」として発表資料には残すが、実装しない。

### 4.1 明示的に除外

| カテゴリ | 除外機能 | 除外理由 |
| --- | --- | --- |
| インフラ | RAG（ベクトル検索） | データ規模が小さくプロンプト直入れで十分 |
| インフラ | データベース（PostgreSQL 等） | JSON ファイルで足りる |
| インフラ | LIFF（LINE Front-end Framework） | 対話 UI で完結できる |
| 学生機能 | 学生同士の直接マッチング | プライバシー設計が必要、MVP 期間で困難 |
| 学生機能 | 他の学生の活動傾向表示 | データ規模が足りず有意な提案が難しい |
| 保護者機能 | 緊急通知 | 判定ロジック・責任範囲の設計が必要 |
| 保護者機能 | AI 利用頻度統計 | 監視感が出やすい、価値が薄い |
| 保護者機能 | サブスク課金機能 | 決済統合が必要、デモには不要 |
| 地域連携 | クーポン・ポイント機能 | 店舗連携の実データが必要 |
| 地域連携 | 地域店舗の自己登録機能 | 架空データ手動投入で代替 |
| 地域連携 | 広告掲載管理 | 収益検証は MVP 対象外 |
| 運営 | モデレーション（投稿承認フロー） | MVP 期間中は AI + 手動で最小限 |
| 運営 | 通報機能 | ユーザー規模が小さく不要 |
| 運営 | 管理者ダッシュボード | 開発者が JSON を直接確認する |
| AI | 医療・法律・緊急対応の判断 | AI 回答制約でオプトアウト、公式窓口誘導 |
| セキュリティ | 個人情報検出・マスキング | 架空データ前提のため対象外 |

### 4.2 発表資料での扱い

除外機能のうち、**プロダクトビジョンとして重要なもの**は発表資料の「今後の展望」セクションで言及する。

- クーポン・ポイントによる地域貢献インセンティブモデル
- 保護者向けサブスク（月額 500 円）による収益モデル
- 地域店舗広告収益モデル
- 学生同士のつながり機能
- 緊急通知セーフティネット

## 5. MVP の成功基準

### 5.1 機能面

- [ ] 学生役アカウントで、プロフィール登録 → やりたいこと相談 → 活動提案受信 → 経験投稿の一連が完走する
- [ ] 学生役アカウントで、生活相談を送信し、先輩投稿を活用した回答が返る
- [ ] 保護者役アカウントで、招待コードを入力し学生と連携できる
- [ ] 保護者役アカウントで、月次サマリーを閲覧し学生の頑張ったことが表示される
- [ ] Quick Reply / Flex Message が両端末（iOS / Android）で表示崩れなく描画される

### 5.2 品質面

- [ ] Webhook 応答時間が平均 8 秒以内、最大 30 秒以内
- [ ] 主要操作 20 回連続で例外・エラーが発生しない
- [ ] `.env` が Git にコミットされていない
- [ ] 実在の学生・保護者情報が含まれていない
- [ ] AI 回答に医療・法律の断定的判断が含まれていない
- [ ] seed カバー外の質問で、捏造された固有名詞・電話番号・日時が回答に含まれていない（NFR-Truth-1）
- [ ] Zero-context 応答で `ZERO_CONTEXT_DISCLAIMER` が必ず含まれる（NFR-Truth-2、`06_ai_spec.md §5.3`）
- [ ] Zero-context + 医療系キーワード検出時に #7119 誘導 (`MEDICAL_FOLLOWUP`) が含まれる（NFR-Truth-3）

### 5.3 プレゼン面

- [ ] デモシナリオ（`08_demo_scenario.md`）を最初から最後まで実演できる
- [ ] 想定質問（`00_product_context.md` 第 16 章）に答えられる
- [ ] SECI モデルによる知識継承の仕組みを 1 事例で説明できる
- [ ] 「JSB の学び・成長・つながり」との接続を口頭で説明できる

## 6. 段階的リリース戦略

3〜4 日という短期間で MVP を完成させるため、機能を段階的にリリースする。

### 6.1 Day 1（Foundation）

**目標**: 「Bot が Webhook を受けて応答を返す」土台を作る

- FastAPI プロジェクト初期化
- LINE Messaging API 接続確認
- Gemini API 接続確認
- 役割選択（学生 / 保護者）フロー
- ヘルプ表示
- systemd + Apache 設定、Webhook URL 登録

### 6.2 Day 2（Core Features）

**目標**: 学生の中心機能を実装する

- 学生プロフィール登録
- やりたいこと相談 + AI 提案
- 生活相談 + 先輩投稿活用
- Flex Message テンプレート整備
- 架空データ 30 + 15 + 10 + 20 件の投入

### 6.3 Day 3（Family Loop）

**目標**: 保護者連携を実装し、家族ループを完成させる

- 招待コード発行・入力フロー
- 経験投稿機能
- 月次サマリー生成
- 保護者向け Flex Message

### 6.4 Day 4（Polish & Demo Prep）

**目標**: デモに耐える品質にする

- Flex Message の見た目調整
- Quick Reply / リッチメニュー整備
- エラーハンドリング強化
- デモシナリオの実機通し
- 発表資料との齟齬解消

### 6.5 スコープ縮退の順序

期間が逼迫した場合は、以下の順で機能を落とす。

1. リッチメニュー（P1）
2. プロフィール参照・更新（P1）
3. 会話コンテキスト短期記憶（P2）
4. Flex Message → テキスト応答に切り替え（見た目の質を下げる）
5. 経験投稿機能 → 事前投入データのみで動かす
6. 月次サマリー → 静的テンプレート化

## 7. MVP 完了後の展望

MVP プレゼン後、以下の順で機能追加を検討する。ただし本ドキュメントの対象外。

1. モデレーション（承認フロー・通報）
2. RAG による回答精度向上
3. 学生同士のマッチング
4. 保護者向けサブスク（決済統合）
5. 地域店舗の自己登録
6. クーポン・ポイント機能
7. 多地域対応
8. 本番運用に向けた法務・プライバシー整備
