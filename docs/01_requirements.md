# 01. 要件定義

## 1. このドキュメントの目的

本ドキュメントは AI 寮母 MVP の**要件定義**を確定するためのものである。プロダクトの目的、解決する課題、機能要件、非機能要件、制約条件を明示的に記述する。

MVP スコープの詳細は `02_mvp_scope.md` に、実装の具体は `04_functional_spec.md` 以降に委ねる。

参照元: `00_product_context.md`（プロダクト企画・背景の完全版）

## 2. プロダクトの目的

### 2.1 プロダクト定義

AI 寮母は、学生マンションに埋もれている先輩の経験・地域情報・行動機会を AI で形式知化し、以下を実現する LINE Bot である。

1. 初めての一人暮らしをする学生の**生活面の不安を解消する**
2. 「何かやりたいけど、何をすればよいかわからない」学生に**学び・成長・つながりのきっかけを提供する**
3. プライバシーに配慮した**保護者向け安心レポート**を提供する

### 2.2 プロジェクトゴール（MVP フェーズ）

- 2026-07-08 頃までに、**同志社大学ビジネスアイデア大会 2026 決勝プレゼンで実演可能な LINE Bot** を完成させる。
- 審査員が実際に触れる状態にする（QR コードで友だち追加 → 主要機能を体験）。
- 学生役・保護者役の 2 アカウントを用いた**デモシナリオが最初から最後まで完結**する。

### 2.3 プロジェクト非ゴール（MVP フェーズ）

- 本番運用に耐えるスケーラビリティ・可用性の確保
- **学生・保護者・先輩投稿の実在情報の取り込み**（プライバシー観点、seed は架空維持）
- モデレーション・審査承認フロー
- 商用リリース水準のセキュリティ監査
- 多地域対応（**同志社大学今出川キャンパス周辺**に限定。京都府全域や京田辺キャンパスは MVP 外）

なお、地域情報・イベント・店舗の seed は同志社今出川周辺の**実在情報ベース**で投入する（`.codex/rules/project_rules.md` の「データ運用」節参照）。「実在情報の取り込み」が非ゴールなのは人物系（学生・保護者・先輩）のみ。

## 3. 解決したい課題

`00_product_context.md` の第 4 章から、MVP で扱う課題を以下 3 つに絞る。

### 3.1 課題 A: 「やりたいことが見つからない」学生の停滞（**主軸**）

学生は「何かやりたい」「成長したい」「地域や人とつながりたい」と考えているが、以下の理由で行動に移せない。

- 自分に合う活動やイベントを探す方法がない
- 周囲の学生が何をしているか見えない
- 行動を起こすきっかけがない

**MVP での解決アプローチ**: 学生プロフィール（大学、興味関心、目標、最近頑張っていること等）を AI が把握し、地域イベント・ボランティア・店舗・活動機会を Flex Message で提案する。

### 3.2 課題 B: 初めての一人暮らしの生活不安（**副軸**）

初めて一人暮らしをする学生は、体調管理、行政手続き、ゴミ出しルール、地域の生活インフラなどで困る。特に夜間・休日に相談先がない。

**MVP での解決アプローチ**: 先輩入居者の投稿と地域情報を活用し、地域・マンションに特化した回答を AI が返す。

### 3.3 課題 C: 保護者の見えない不安（**副軸**）

保護者は遠方の子どもの生活状況が見えず、過干渉にならない範囲での安心材料を求めている。

**MVP での解決アプローチ**: 学生本人の同意のもと招待コード方式で連携し、月次サマリー（学生の「頑張ったこと」）を保護者に届ける。

### 3.4 MVP で扱わない課題

`00_product_context.md` で言及されている以下の課題は、MVP には含めず「今後の展望」として発表資料に残す。

- 地域店舗の広告収益モデル（クーポン・ポイント）
  - ※企業スポンサードイベントの PR 掲載（FR-S9）は別収益源として MVP に含む（`02_mvp_scope.md §3`、`00_product_context.md §10.7`）。課金・入稿管理は対象外。
- 学生同士の直接マッチング
- 地方創生・文化継承への貢献
- JSB の物件価値向上・入居率向上の検証
- モデレーション・投稿確認フロー

## 4. ステークホルダーと提供価値

### 4.1 主要ステークホルダー

| ステークホルダー | MVP における位置付け |
| --- | --- |
| 学生（利用者） | 主要ユーザー。プロフィール登録・生活相談・活動提案を受ける。 |
| 保護者（利用者） | 副次ユーザー。学生との連携後、月次サマリーを受け取る。 |
| 審査員（評価者） | デモ体験を通じて MVP の価値を評価する。 |
| 提案チーム（開発者） | MVP 実装・発表・運用を担う。 |

### 4.2 MVP で切り出す提供価値

| ステークホルダー | 提供価値（MVP 範囲） |
| --- | --- |
| 学生 | やりたいこと相談による活動提案 / 生活相談への地域特化回答 / 経験投稿による貢献感 |
| 保護者 | 招待コードによる連携 / 月次「頑張ったこと」サマリー閲覧 |
| 審査員 | 実機での機能体験 / SECI モデルに基づく知識継承の仕組みの実感 |

### 4.3 MVP で対象外のステークホルダー

- JSB（企業側の意思決定者）: 発表資料では触れるが、システム上のロールは実装しない
- 地域店舗・地域団体: 実在情報を手動で seed 投入、自己登録機能は実装しない
- 地方自治体・地域団体: 対象外

## 5. 機能要件（High-Level）

具体仕様は `04_functional_spec.md` を参照。ここでは要件レベルで列挙する。

### 5.1 学生向け機能

| ID | 機能 | 概要 |
| --- | --- | --- |
| FR-S1 | 役割選択 | 初回友だち追加時に「学生」を選択できる |
| FR-S2 | プロフィール登録 | 大学・学部・学年・興味関心・目標・最近頑張っていることを対話形式で登録できる |
| FR-S3 | プロフィール参照・更新 | 登録済みプロフィールを確認・更新できる |
| FR-S4 | やりたいこと相談 | AI が学生プロフィールと地域データから 2〜3 件の活動を Flex Message で提案する |
| FR-S5 | 生活相談 | 生活の困りごとを AI に相談でき、先輩投稿・地域情報・他の学生の経験投稿（匿名化）を反映した回答が返る |
| FR-S6 | 経験投稿 | 参加したイベント、利用した店舗、生活の知恵を対話形式で投稿できる |
| FR-S7 | 保護者連携 | 6 桁の招待コードを発行し、保護者との紐付けができる |
| FR-S8 | 頑張ったこと記録 | 経験投稿の一部が「頑張ったこと」として保護者向けサマリーに反映される |
| FR-S9 | スポンサーPR掲載 | やりたいこと相談の活動提案に、学生プロフィールに合致した協賛企業イベント（就活・ハッカソン等）を PR 枠として掲載し、明示表示とクリック計測を行う（`04_functional_spec.md §4.3`） |

### 5.2 保護者向け機能

| ID | 機能 | 概要 |
| --- | --- | --- |
| FR-P1 | 役割選択 | 初回友だち追加時に「保護者」を選択できる |
| FR-P2 | 学生連携 | 学生が発行した 6 桁の招待コードを入力して紐付けができる |
| FR-P3 | 月次サマリー閲覧 | 「今月のレポート」ボタンで、連携している学生の頑張ったことを Flex Message で確認できる |

### 5.3 共通機能

| ID | 機能 | 概要 |
| --- | --- | --- |
| FR-C1 | ヘルプ | 使い方の案内を表示できる |
| FR-C2 | 役割の再選択 | 誤って選択した場合、役割を切り替えられる |
| FR-C3 | Quick Reply / Flex Message | 主要操作はテキスト入力だけでなくボタン操作で完結できる |

## 6. 非機能要件

### 6.1 パフォーマンス

- LINE Webhook からの応答は 30 秒以内（LINE 側タイムアウト制約）
- 通常の応答時間は 3 〜 8 秒以内を目安（Gemini API 呼び出しを含む）

### 6.2 可用性

- MVP 期間中は「発表当日に動く」ことを最優先とする
- 24/365 稼働は保証しない（自宅サーバー故障時は代替なし）
- systemd で自動再起動を設定し、プロセスクラッシュから回復する

### 6.3 セキュリティ・プライバシー

- LINE user_id はハッシュ化してログに出す（生の ID はログ出力しない）
- 学生・保護者・先輩投稿の seed は架空を維持し、実在人物の個人情報を扱わない
- 地域・イベント・店舗の seed は実在情報ベース（同志社今出川周辺）。民間店舗については情報鮮度注記を全レコード必須付与する
- API キー・チャネルトークンは `.env` 経由で管理、リポジトリにコミットしない
- 保護者に共有される情報は「学生本人が同意した範囲」に限定
- 医療・法律・緊急対応は AI が断定せず、公式窓口へ誘導する

### 6.4 AI 回答の信頼性（Truthfulness）

Gemini が seed に無い固有情報を捏造することを防ぐため、以下を非機能要件として明記する。詳細な実装仕様は `docs/06_ai_spec.md §5.3`、機能フローは `docs/04_functional_spec.md §4.4` を参照。

- **NFR-Truth-1**: 情報源（`data/seed/*.json`）に存在しない固有名詞・電話番号・営業時間・日時を回答に含めない。
- **NFR-Truth-2**: 情報源にヒットが 0 件の場合、応答冒頭に情報源の限界を示す disclaimer を付与し、公式窓口へ誘導する（Zero-context 分岐）。
- **NFR-Truth-3**: 医療系相談（「病院」「クリニック」「熱」「体調」等の非緊急キーワード）で情報源が不十分な場合、京都府救急安心センター（#7119）または京都市の医療機関検索サイトへの誘導を必ず含める。
- **NFR-Truth-4**: seed が実在情報の場合（地域・イベント・店舗）、Bot は必ず情報鮮度注記を併記する。店舗 seed の `data_freshness_note` フィールドの内容、または「◯年◯月時点の情報のため、営業状況等が変化している可能性がある」旨を Flex Message に必ず含める。

### 6.5 保守性

- コードは Python 型ヒント必須、Google スタイル docstring
- Black + Ruff でフォーマット・リント
- モジュール分割で 1 ファイル 500〜700 行以内
- 主要な設計判断は `docs/` に記録

### 6.6 デプロイ・運用

- 自宅 Ubuntu サーバー上で systemd 管理
- Apache リバースプロキシ配下（`linebot.kmchan.jp/ai_house_mother`）
- ログは journald 経由で確認可能
- 再起動: `sudo systemctl restart ai_house_mother.service`

## 7. 制約条件

### 7.1 期間

- **開発期間**: 3 〜 4 日（〜 2026-07-08 頃）
- **発表**: 2026-07-XX（決勝プレゼン日、確定次第 `00_product_context.md` に反映）

### 7.2 予算

- LLM API コストは最小限に抑える（Gemini 2.0 Flash-Lite 選定理由）
- 有料インフラの新規契約は行わない（既存の自宅サーバーで完結）

### 7.3 技術

- Python 3.12 + FastAPI + line-bot-sdk v3 に固定
- LLM は Gemini 2.0 Flash-Lite に固定
- データ永続化は JSON ファイル（DB 導入なし）
- RAG は実装しない
- LIFF は使わない

### 7.4 チーム

- MVP 実装は主に緑川（ユーザー）が担当
- 引き継ぎフェーズ以降は安樂等が参加する可能性がある
- ドキュメント・規約は他メンバーが読んで理解できる状態にする

### 7.5 法務・プライバシー

- MVP はデモ・発表用のため、本番運用に必要な同意フロー・規約は最小限
- ただしデモ中でも「学生・保護者・先輩の実在個人情報を扱わない」ことを徹底（地域・イベント・店舗の実在情報は問題なし）
- 発表資料には「MVP はデモ用途で、本番運用時には法務確認が必要」と明記

## 8. 仮定と依存

### 8.1 前提となる仮定

- 発表当日にインターネット接続とサーバー稼働が確保される
- Gemini API が利用可能で、レートリミット制約に収まる
- LINE Messaging API のサンドボックスまたは公式アカウントが取得できる

### 8.2 外部依存

- **LINE Developers Console**: チャネル作成、Webhook URL 登録
- **Google AI Studio**: Gemini API キー取得
- **自宅 Ubuntu サーバー**: Apache・systemd・ネットワーク

### 8.3 リスクと軽減策

| リスク | 影響 | 軽減策 |
| --- | --- | --- |
| Gemini API のレートリミット・障害 | 応答不可 | エラー時のフォールバック文言を用意 |
| 自宅サーバーの停電・接続断 | デモ実施不可 | 予備回線・オフライン録画バックアップ |
| LINE 公式アカウント取得の遅延 | Bot 実機テスト不可 | 早期取得を最優先タスクにする |
| 実装期間の逼迫 | 機能未完成 | スコープを段階的に切り詰める（`02_mvp_scope.md` 参照） |
