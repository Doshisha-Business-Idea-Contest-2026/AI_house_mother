# 08. デモシナリオ

## 1. このドキュメントの目的

決勝プレゼン当日のデモの操作手順を定義する。

- 事前準備
- 使用アカウントとデバイス
- 見せる機能とその順序
- 想定質問と回答
- タイムラインと保険

## 2. デモの狙い

- **主軸**: 「やりたいことが見つからない学生に、AI が具体的な行動きっかけを提示する」体験を見せる
- **副軸**: 「保護者が過干渉にならずに子どもの頑張りを見守れる」設計を見せる
- **差別化**: 汎用 AI ではなく、「その地域・そのマンションに特化した回答」ができる仕組みを見せる（先輩投稿の活用）

## 3. 事前準備

### 3.1 アカウント・デバイス

| アカウント | デバイス | 役割 | LINE 表示名 |
| --- | --- | --- | --- |
| 学生役 | スマートフォンA | 「山田 春樹」（P-S1） | 山田 春樹 |
| 保護者役 | スマートフォンB | 「山田 由美」（P-P1、春樹の母） | 山田 由美 |

- 両アカウントとも、事前に AI 寮母を友だち追加する
- 発表直前に **アカウントとセッションをリセット**（`data/users.json` から該当ユーザーを削除、キャッシュクリア）することで、審査員の目の前で「初期状態からのオンボーディング」が見えるようにする

### 3.2 サーバー状態

- `data/seed/*.json` は完全に投入済み（地域 30、店舗 15、イベント 10、先輩投稿 20）
- `data/users.json` / `profiles.json` / `posts.json` / `invitations.json` / `parent_links.json` はデモ直前に空に戻す
- `systemctl status ai_house_mother.service` で稼働確認
- `curl https://linebot.kmchan.jp/ai_house_mother/health` で疎通確認
- `.env` の `LOG_LEVEL=INFO` に設定、journald で応答を確認できる状態にしておく
- Gemini API のクレジット残量を確認

### 3.3 発表資料との連携

- スライドにデモ画面のスクリーンショットを保険として用意（当日ネット不通時のバックアップ）
- LINE のミラーリング環境を整備（スマホ画面をプロジェクター/大画面に表示）
- QR コード（審査員が触る用）をスライドまたは配布資料に印刷

## 4. デモの流れ

### 4.1 全体構成（想定 5〜7 分）

```
[0:00] 発表冒頭 → プロダクト概要（スライド）
[0:30] 課題の説明（スライド）
[1:00] AI 寮母の提案（スライド）
[2:00] デモ開始
       ├── [2:00] 学生役: 友だち追加 → 役割選択 → プロフィール登録
       ├── [3:00] 学生役: やりたいこと相談 → AI が活動提案
       ├── [3:45] 学生役: 招待コード発行 → 保護者役に共有
       ├── [4:00] 保護者役: 友だち追加 → 役割選択 → コード入力 → 連携完了
       ├── [4:45] 学生役: 生活相談（発熱） → 先輩投稿を反映した回答
       ├── [5:15] 学生役: 経験投稿（さっき提案された活動に参加した）
       └── [5:45] 保護者役: 今月のレポート → 学生の頑張り確認
[6:00] 差別化ポイントと今後の展望（スライド）
[7:00] 質疑応答
```

### 4.2 詳細スクリプト

#### シーン 1: 学生の初回体験（2:00〜3:00）

**発表者の話**: 「学生の LINE 画面をご覧ください。AI 寮母を友だち追加します。」

**操作**:
1. スマホA で AI 寮母を友だち追加
2. Bot からウェルカム + 役割選択が届く
3. 「👨‍🎓 学生です」を選択
4. プロフィール登録の案内 → 「✍️ プロフィールを登録する」
5. 対話形式で以下を入力（事前に台本メモ）:
   - 大学: 同志社大学
   - 学部: 経済学部
   - 学年: 1
   - 興味: 地域活動、ものづくり、食・カフェ巡り
   - 最近頑張っていること: TOEIC の勉強
   - やってみたいこと: 京都の伝統文化に触れたい

#### シーン 2: やりたいこと相談（3:00〜3:45）

**発表者の話**: 「大学 1 年生が『何かやりたいけど、何をすれば良いかわからない』ときに、AI 寮母がどうサポートするかご覧ください。」

**操作**:
1. メニューから「🎯 やりたいこと相談」
2. Bot が数秒考える表示
3. Flex Message カルーセルで 2〜3 件の活動提案が表示される
   - 期待する内容:
     - 下鴨神社の月例清掃ボランティア
     - 進々堂 京大北門前での勉強会企画
     - 地域文化イベント（京都の伝統に触れる）
4. 学生役が 1 つのカードで「詳しく聞く」を押す
5. Bot が追加情報を返す

**発表者の話**: 「注目していただきたいのは、Gemini がプロフィールと地域データを組み合わせて、この学生特有の提案を返している点です。汎用 AI では京都・同志社周辺の情報も、学生の興味も踏まえた提案はできません。」

#### シーン 3: 招待コード発行（3:45〜4:00）

**発表者の話**: 「学生から保護者に、招待コードで連携できる仕組みも用意しています。」

**操作**:
1. メニューから「👨‍👩‍👧 保護者連携」
2. Bot が 6 桁のコードを返す（例: `A3F7K9`）
3. 発表者が「このコードを保護者に送ります」と口頭で説明

#### シーン 4: 保護者の連携（4:00〜4:45）

**発表者の話**: 「今度は保護者役の LINE 画面です。」

**操作（スマホB）**:
1. AI 寮母を友だち追加（Bot からウェルカム）
2. 「👨‍👩‍👧 保護者です」を選択
3. 「🔑 コードを入力する」
4. 先ほどの 6 桁コードを入力
5. Bot が「山田 春樹さんと連携完了」と応答
6. 学生役のスマホA には「保護者と連携しました」の通知

**発表者の話**: 「保護者は招待コードを入力するだけで、監視ではなく合意ベースの見守りが始まります。」

#### シーン 5: 生活相談（4:45〜5:15）

**発表者の話**: 「学生が実際に困ったときも AI 寮母が寄り添います。」

**操作（スマホA）**:
1. テキストで「熱っぽい、近くの病院ある？」と送信
2. Bot が回答（先輩投稿を活用した内容）:
   - 例: 「近くだと今出川の◯◯クリニックが便利です。先輩の体験より: 『土曜午前も空いていて待ち時間が短かった』…」
   - 「症状が続く場合は医療機関を受診してください」の注記

**発表者の話**: 「Gemini の回答に、先輩入居者が投稿した情報が反映されています。これが SECI モデルの『暗黙知を形式知にして次の学生に継承する』仕組みです。」

#### シーン 6: 経験投稿（5:15〜5:45）

**発表者の話**: 「学生が実際に活動に参加したあと、経験を投稿することで次の学生に価値が渡ります。」

**操作（スマホA）**:
1. 「✏️ 経験を投稿」
2. カテゴリ: 🧹 ボランティア
3. タイトル: 下鴨神社の清掃活動に参加した
4. 本文: 「先週の土曜日、下鴨神社の清掃に参加してきました。地域の方と話せて楽しかったです。」
5. 場所: 下鴨神社
6. 「頑張ったこととして保護者に共有しますか？」→ はい
7. 投稿完了

#### シーン 7: 保護者の月次レポート（5:45〜6:00）

**発表者の話**: 「保護者はどう見守るか。」

**操作（スマホB）**:
1. 「📊 今月のレポート」（Pull 主軸）
2. Bot が Flex Message で当月の頑張ったことを表示
3. 発表者が画面を見せながら「会話内容や生活相談は共有されず、学生本人が『共有したい』と選んだ頑張りだけが届きます」と説明

**発表者の補足**: 「実運用では毎月 1 日の朝 9 時に、自動でこのレポートが保護者へ配信されます。今回のデモではその場で保護者が Pull した形をお見せしていますが、Push は systemd timer で常時走っています。」

**Push デモを見せる場合の手動実行手順**（任意、時間に余裕がある場合）:

```bash
# 前月 (2026-06) 分の Push を全連携保護者に強制送信
python scripts/push_monthly_reports.py --month 2026-06 --force
```

- `--force` を付けると `data/monthly_report_state.json` の記録があっても再送される。
- `--now` を指定すると架空の実行時刻を渡せる（月境界判定の再現テスト用）。
- デモ当日は Pull を主軸で見せ、Push は口頭説明のみで十分。

## 5. 差別化ポイントの強調

デモ中に必ず口頭で述べるべき差別化ポイント:

1. **地域特化**: 「京都・同志社に特化した回答」— 汎用 AI にはできない
2. **暗黙知の形式知化**: 「先輩の体験が次の学生に継承される」— SECI モデル
3. **保護者との合意ベース**: 「監視ではなく、学生本人の選択で共有される」
4. **一つの LINE Bot で 3 者価値提供**: 学生・保護者・JSB（さらに将来的に地域社会）
5. **本人の意志で成長機会をつかむ**: 「何かやりたいけどわからない」から「これを試してみる」へ

### 5.1 実装で見せる 3 つの磨き込み（口頭補足の余地）

デモ中の待機や画面遷移で「これは何が動いているのか」に触れる余地があるとき、以下 3 点を軽く補足すると差別化が伝わる:

- **Loading Indicator（T4.11）**: 「Gemini 呼び出し中は LINE 標準の入力インジケーターを出しているので、ユーザーは中間テキストではなく自然な待機で結果を受け取れます」
- **Sender switch（T4.1b）**: 「送信者のアイコンと名前で AI 寮母の会話 / システム応答 / 通知を切り替え、UX の切れ目を明示しています」
- **SECI 継承（T4.10）**: 「学生の経験投稿は匿名化した上で後の学生の生活相談 Gemini プロンプトに参照されます。つまり、この LINE の中で暗黙知が形式知になり、次の後輩に継承されている実装レベルで動いています」

## 6. 想定質問と回答

`00_product_context.md` 第 16 章および決勝向けブレストから、質問候補と回答を準備する。

### Q1. 汎用 AI（ChatGPT、Gemini そのまま）で十分では？

**回答**: 汎用 AI はマンション固有の情報、先輩の実体験、地域特化のディープな情報を持っていません。AI 寮母は「学生自身が情報提供者になる」設計で、その暗黙知を AI に継承させます。

**実装レベルの裏付け**（T4.10 SECI 継承）: 学生が「✏️ 経験を投稿」で残した知恵は、匿名化した上で（`title` / `body` / `area` / `category` / `created_at` のみ）その後の学生の生活相談 Gemini プロンプトに参照情報として渡されます。投稿者の LINE userId・プロフィール・`post_id` は絶対に伝播しません。汎用 AI では「その学生マンションの先輩が実際に投稿した知恵」に基づいた回答は原理的に得られません。詳細は `docs/06_ai_spec.md §4.2` と `docs/05_data_model.md §8`。

### Q2. 学生が本当に情報を投稿するインセンティブは？

**回答**: MVP では「後輩の役に立つ」貢献感と「保護者への頑張り共有」を主軸にしています。今後の展望として、地域店舗のクーポン付与によるインセンティブモデルも設計しています。

### Q3. 保護者が監視ツールとして使うリスクは？

**回答**: 招待コード方式で学生本人が発行しないと連携できず、共有内容も学生本人が「共有する」と選んだ投稿のみに限定しています。会話ログや生活相談内容は保護者に届きません。

### Q4. プライバシー・個人情報の扱いは？

**回答**: MVP はデモ用で架空データのみを使っています。本番運用時は法務・専門家に確認して、同意フロー・データ削除ポリシーを整備する必要があると認識しています。

### Q4.5. ハルシネーション対策は？（NFR-Truth-1/2/3）

**回答**: AI 寮母は 3 段構えでハルシネーションを抑えています。

1. **プロンプトレベル**: Gemini に「情報源にない固有情報（電話番号・営業時間・特定店舗名）を断定しない」制約を渡す（`docs/06_ai_spec.md §3` の System Prompt）。
2. **コード判定レベル (Zero-context)**: 情報源（seed の 3 コレクション + 学生投稿）に該当が 0 件のとき、応答冒頭で `ZERO_CONTEXT_DISCLAIMER` を必ず付与し、京都市エコまちステーションや #7119 京都府救急安心センターなど公式窓口への誘導に切り替える（`docs/06_ai_spec.md §5.3`）。
3. **情報源の拡張レベル (T4.10 SECI 継承)**: 学生の経験投稿を匿名化して生活相談 Gemini の参照情報に含めることで、seed に無いローカルな話題（例: 上京区でのダンボール回収の裏技）にも「同じマンションの先輩の投稿より」と正確に引用できる状態になりました。

3 段のいずれかが必ず作動するため、seed に無い質問でも「知らないふりをして情報源から作文する」失敗を防いでいます。非機能要件は `docs/01_requirements.md §6.4`、詳細実装は `docs/06_ai_spec.md §4.2 / §5.3`。

**補足デモ案**: シーン 5（生活相談・発熱）は seed カバー内で通常応答が返るため、Zero-context が発動しません。審査員が seed カバー外の質問（例: 「〇〇区のゴミ収集日を教えて」など）を投げた場合、`ZERO_CONTEXT_DISCLAIMER` が表示されるセーフティネットの実物をその場で見せられます。逆に「上京区でダンボール捨てたい」のように過去に学生が投稿した内容と近い質問なら、T4.10 SECI 継承で「学生の投稿より」と匿名引用される様子を見せられます。想定外の質問こそが、このプロダクトの誠実さを示すデモ機会になります。

### Q5. どうやって収益化するのか？

**回答**: 主軸は保護者向けの月額サブスクリプション（月 500 円想定、学生数を仮に 3 万人とすると年間 1.8 億円）と、地域店舗からの掲載料モデル（月 2000 円 × 2000 店舗 = 年 4800 万円）です。合わせて年 2.28 億円規模の可能性があります。

### Q6. 収益は誰のものか？ JSB との関係は？

**回答**: 学生マンション事業者である JSB と協業する想定で、JSB の物件価値向上（差別化・入居率向上・管理業務効率化）と併せて、AI 寮母事業自体の収益をシェアするモデルを検討しています。

### Q7. 他社にすぐ真似されないか？

**回答**: 短期的には UI・機能は真似できますが、AI 寮母の本質は「学生マンション固有の知識資産」の蓄積です。運用年数を経ることで、他社が模倣できない情報資産に育ちます。

### Q8. 開発体制と期間は？

**回答**: MVP は宮崎ゼミの学生 6 名で 3〜4 日で構築しました。使用技術は Python + FastAPI + LINE Bot + Google Gemini。

## 7. リスク・保険

### 7.1 当日の技術トラブル

| リスク | 影響 | 保険 |
| --- | --- | --- |
| ネット不通 | Bot 応答不能 | スクリーンショットスライドで代替 |
| Gemini API 障害 | AI 回答不能 | 事前録画動画で代替 |
| LINE 側障害 | 実演不可 | 上記スクリーンショット + 事前録画 |
| サーバー停電 | 実演不可 | 上記スクリーンショット + 事前録画 |

**推奨**: デモの実機実行を第一に、事前録画とスライド SS を第二保険として用意。

### 7.2 想定外の質問への対応

- 「わかりません」と正直に答える
- 「持ち帰って検討します」で保留する
- 立てた仮定・未確定事項（`00_product_context.md` 第 17 章）を隠さない

### 7.3 デモの想定通りに動かない場合

- 焦らず落ち着いて口頭で説明を続ける
- 「実装上のバグですが、コンセプトは以下の通りです」と展開する
- 事前録画動画への切り替えを躊躇わない

## 8. 発表後の運用

### 8.1 審査員が触れる場合

- QR コード配布 → 友だち追加 → 実際に触ってもらう

- 発表後、審査員の LINE user_id を含むレコードを `data/` から削除
- 再度デモの機会があれば `data/users.json` 等をリセット
