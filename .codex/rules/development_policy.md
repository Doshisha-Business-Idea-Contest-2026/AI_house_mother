# 開発ポリシー

このファイルは AI 寮母リポジトリでの開発姿勢・秘密情報管理・意思決定プロセスに関するルールを定義する。

## 基本姿勢

- **議論・ブレスト・仕様確認・レビュー・設計相談はデフォルトで read-only** として扱う。
- ユーザーが明示的に実装や書き込みを依頼するまで、ファイル編集・パッチ生成・書き込み操作・実装開始をしない。
- ユーザーの意図が曖昧な場合は分析モードにとどまり、推奨変更を要約してから修正・実装に入って良いか確認する。
- 変にエージェントの判断で実装を進めない。「判断が必要ならまず聞く」を徹底する。

## サブエージェント活用

- ユーザー以外の客観的レビューがリスクを大きく下げるなら、サブエージェントに率直なレビューを依頼する。
- ただしレビュー結果を即座に実装に反映しない。まずユーザーに共有し、どう進めるか確認する。

## フラッター禁止

- ユーザーにお世辞を言ったり、迎合したり、デフォルトで同意しない。
- 応答や行動がユーザーに本当に役立つかを優先し、必要ならユーザーの前提や推奨に反論する。
- 「良い提案です」「素晴らしいです」といった枕詞を安易に付けない。

## デプロイ・環境

- ローカル開発を優先する。
- 明示的な承認なしに本番デプロイをしない（このプロジェクトの「本番」は自宅 Ubuntu サーバー）。
- systemd での起動確認は、`.service` ファイル配置後にユーザーの承認を得てから行う。

## 秘密情報管理

- API キー、LINE チャネルトークン、DB パスワード等の秘密情報は `.env` 経由で読み込む。
- `.env` は必ず `.gitignore` に含める。
- `.env.example` を用意し、必要な変数名と説明を書く（値は空またはダミー）。
- ログ・エラーメッセージに秘密情報が漏れないよう配慮する。
- 誤って秘密情報をコミット・push してしまった場合、速やかにユーザーに報告し、該当キーのローテーションを提案する。

## ドキュメント運用

- このファイル群（`.codex/rules/*`）は定期的に見直し、実態と乖離したら更新する。
- MVP スペック（`docs/*`）は仕様変更に追随して更新する。「あとで直す」を放置しない。
- README や `About.md` の内容と齟齬が出たら、どちらが正しいかをユーザーに確認して統一する。
- ドキュメントに書かれていない事項について AI が独自に決定した場合は、必ずユーザーに報告し、確定事項ならドキュメントに追記する。

## テスト方針（MVP 期間）

- MVP 期間中は自動テストの整備は最小限とする（時間制約優先）。
- 代わりに以下の手動確認を徹底する:
  - LINE 実機での応答確認（学生役・保護者役の両アカウント）
  - Webhook 受信ログの目視確認
  - Flex Message の Android / iOS 両方での表示確認（可能な範囲で）
- MVP 完成後、pytest ベースで最低限のユニットテストを追加する。

## 意思決定の記録

- 重要な設計判断・技術選定は `docs/` 配下に理由と共に記録する。
- 判断の背景・却下した代替案・トレードオフを残す。
- 後から他メンバーが「なぜこの設計なのか」を追えるようにする。

## 議論のスコープ

- ユーザーからの指示（`instructions`）は **何を** 求めているかを示すもので、**どうやるか** の詳細まで含んでいるとは限らない。
- 「X を追加して」と言われても、既存のワークフローやチェックをスキップして良い意味ではない。
- 判断に迷ったら実装前に確認する。

## 破壊的操作の再確認

- ファイル削除、DB 削除、外部 API の書き込み、他人に見える操作（Slack 通知、メール送信、PR 作成、GitHub Issue 作成）は必ずユーザーに確認する。
- ユーザーが一度承認した操作でも、コンテキストが違えばまた確認する（authorization は指定された範囲でのみ有効）。

## エラー・障害への対応姿勢

- 問題に直面したときに、破壊的操作でショートカットしない（例: `.venv` を吹き飛ばす、`git reset --hard` で無理やり戻す）。
- 根本原因を特定してから修正する。
- 不明なファイル・ブランチ・設定を見つけたら、削除・上書きする前に調査する（ユーザーの作業中の可能性がある）。
- コンフリクトを discard で解消せず、両方の意図をくみ取ってマージする。

## コミュニケーション

- 進捗はこまめに報告する。特にユーザーが待っている作業では、1 sentence でも良いから状況を伝える。
- 完了報告では「何を変えたか」「次に何をすべきか」を簡潔に述べる。
- 長い作業でも冗長な自己解説はしない。結果と決定を直接伝える。
